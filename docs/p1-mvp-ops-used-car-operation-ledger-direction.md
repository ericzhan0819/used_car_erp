## 1. 本文件目的

本文件正式定義 Used Car ERP MVP 的方向調整。

原本主線逐步建立：

```text
車輛頁業務操作
→ Used Car Money Flow
→ Used Car Voucher Draft
→ Journal Entry / Sales Invoice
→ ERPNext 會計閉環
```

經重新檢視小型中古車行創業初期的實務需求、資金限制、紙本文件保存流程與記帳士分工後，MVP 主線調整為：

```text
車行營運管理帳
+ 資金帳
+ 單車損益
+ 文件檢查
+ 成交結案列印
+ 記帳士交接包
```

核心結論：

```text
Used Car ERP 現階段不是要取代記帳士，也不是要自動完成正式會計與報稅。
Used Car ERP 現階段要先讓小型中古車行每天營運不混亂。
```

## 2. 最終定位

`used_car_erp` 仍是 ERPNext custom app，但 MVP 定位收斂為：

```text
ERPNext 上的中古車行營運操作層。
```

更具體地說，它是：

```text
小型中古車行的營運管理帳與紙本資料索引系統。
```

不是：

```text
完整客製 ERP
完整會計系統
自動報稅系統
記帳士替代品
大型車商管理平台
多租戶 SaaS
```

---

## 3. 業務現實背景

目前車行實務預設：

1. 每台車仍建立紙本資料夾。
2. 買賣合約、讓渡書、發票、收據、匯款證明、監理文件等正式文件，仍以紙本逐台保存。
3. ERP 不需要保存所有正式文件原件。
4. ERP 需要做營運資料記錄、統計、查詢、提醒與彙整。
5. 成交後，ERP 應能列印該車「車輛結案明細表」，放入紙本資料夾。
6. 每個申報期，ERP 應整理收入、支出、買入、售出、15-1 與缺件資料，交給記帳士。
7. 正式會計科目、傳票、申報與報稅責任，由記帳士處理或確認。

ERP 與紙本資料夾分工：

| 項目 | 負責方式 |
| --- | --- |
| 正式文件原件 | 紙本資料夾 |
| 營運資料查詢 | ERP |
| 車輛成本與毛利 | ERP |
| 現金與銀行流向 | ERP |
| 文件是否缺漏 | ERP |
| 成交結案明細 | ERP 列印後放入紙本 |
| 申報資料整理 | ERP 匯出 / 列印 |
| 正式會計與報稅 | 記帳士 |

## 4. MVP 主線調整

### 4.1 不再以完整會計閉環作為 MVP 主線

後續 MVP 不再優先投入：

```text
Journal Entry 自動化細節
Sales Invoice 完整串接
Advance Reconciliation
正式會計科目配置
完整借貸分錄
完整財報
自動報稅
```

這些功能不是刪除，也不是永遠不做，而是降級為：

```text
後期功能
會計輔助層
選配功能
非 MVP 驗收重點
```

目前更重要的是：

```text
每台車花多少
每台車收多少
每台車賺賠多少
現金還有多少
銀行還有多少
哪些款項待收
哪些款項待付
哪些文件缺漏
哪些資料要交給記帳士
```

### 4.2 業務頁面不碰正式會計語意

業務人員與日常操作頁不應被迫理解：

```text
借方
貸方
Journal Entry
Sales Invoice
Reconciliation
Advance Account
會計科目
```

業務操作只使用車行語意：

```text
買車
整備
維修
美容
代辦
上架
保留
收訂金
收尾款
退款
成交
取消保留
結案
```

## 5. Money Flow 成為 MVP 主帳

`Used Car Money Flow` 是現階段最重要的資料核心。

它不是正式會計分錄，而是車行營運事實紀錄。

每筆 Money Flow 應能回答：

```text
這筆錢是什麼？
跟哪台車有關？
是收入還是支出？
金額多少？
日期是什麼？
對象是誰？
用現金還是銀行？
是否已結清？
是否有憑證？
是否影響單車成本 / 毛利？
是否需要交給記帳士？
```

後續設計重點應從「如何更完整產生正式會計文件」改為：

```text
如何讓 Money Flow 成為老闆每天可查、可算、可交接的營運主帳。
```

## 6. 三層架構

後續專案以三層架構思考。

### 6.1 第一層：業務事實層

此層負責紀錄車行每天發生的事。

包含：

```text
新增車輛
買入車輛
記錄買入來源
記錄買入成本
記錄整備支出
記錄維修支出
記錄美容支出
記錄代辦支出
記錄其他支出
車輛上架
車輛保留
收訂金
收尾款
退款
成交
取消保留
結案
```

此層禁止要求使用者選擇正式會計科目。

### 6.2 第二層：管理帳層

此層負責老闆每天看得懂的營運結果。

包含：

```text
單車總成本
單車總收入
單車預估毛利
庫存資金占用
現金餘額
銀行餘額
待收款
待付款
本月收入
本月支出
缺憑證筆數
未交記帳士筆數
```

此層是 MVP 核心價值。

### 6.3 第三層：會計交接層

此層不負責自動報稅或正式入帳，而是整理資料給記帳士。

包含：

```text
本期收入明細
本期支出明細
本期買入車輛明細
本期售出車輛明細
本期 15-1 判斷明細
本期缺憑證清單
現金 / 銀行流水摘要
附件 / 紙本文件索引
匯出 CSV / Excel
列印交接表
```

## 8. 15-1 稅務邊界維持

15-1 只用在「售車營業稅估算」。

核心規則：

```text
purchase_price = 購車價
購車價不包含整備、維修、美容、拍場、代辦或其他後續費用
整備 / 維修 / 美容 / 拍場 / 代辦不併入 15-1 購入成本
```

管理損益與 15-1 稅務估算必須分開：

```text
單車管理損益 = 成交價 - 購車價 - 整備 / 維修 / 美容 / 拍場 / 代辦 / 其他直接支出
```

```text
15-1 營業稅估算 = 只看售車價與購車價
```

後續若新增資金帳或結案表，不得把後續支出混入 15-1 購入可扣抵基礎。

## 9. 紙本資料夾與 ERP 的關係

紙本資料夾仍保存正式文件。

可能包含：

```text
買賣合約
讓渡書
普通收據
統一發票
匯款證明
行照影本
監理文件
整備維修單
美容單
代辦文件
客戶簽收文件
其他特殊文件
```

ERP 負責索引與檢查：

```text
文件是否存在
文件類型
文件日期
文件金額
關聯車輛
關聯交易
是否缺件
是否已交記帳士
備註
```

ERP 重點不是保存每份正式文件原件，而是讓紙本資料夾可被查詢、檢查與交接。

## 10. 成交結案明細表

每台車成交後，ERP 應能列印：

```text
車輛結案明細表
```

列印後放入該車紙本資料夾。

建議內容：

| 區塊 | 欄位 |
| --- | --- |
| 車輛資料 | 車輛編號、車牌、廠牌、型號、年份、VIN |
| 買入資料 | 買入日期、買入來源、賣方、買入價格 |
| 成本明細 | 整備、維修、美容、代辦、拍場、其他支出 |
| 銷售資料 | 買方、成交日期、成交價、訂金、尾款、退款 |
| 資金流向 | 現金收付、銀行收付、待收、待付 |
| 稅務提示 | 是否 15-1、是否有發票、是否有買賣合約 / 讓渡書 |
| 毛利摘要 | 成交價、總成本、預估毛利 |
| 文件檢查 | 必要文件是否齊全 |
| 備註 | 特殊狀況、糾紛、折讓、保固約定 |

這是 ERP 與紙本流程的關鍵橋接點。

## 15. 驗收標準

P1-MVP-OPS 的驗收標準不是會計傳票是否完整，而是：

1. 新增一台車後，所有買入、整備、上架、保留、成交資料都能記錄。
2. 每台車能即時看出總成本、總收入、預估毛利。
3. 每筆現金 / 銀行流入流出都能查到來源。
4. 系統能看出現金、銀行、待收、待付。
5. 系統能提示每台車缺哪些文件。
6. 成交後能列印車輛結案明細，放入紙本資料夾。
7. 每期能整理出交給記帳士的收入、支出、買入、售出、15-1 與缺件明細。
8. 業務頁面不出現正式會計技術語意。

## 17. 最終結論

Used Car ERP 不應在創業初期追求完整 ERPNext 會計閉環。

更正確的方向是：

```text
車行營運管理帳
+ 資金帳
+ 單車損益
+ 文件檢查
+ 成交結案列印
+ 記帳士交接
```

紙本資料夾仍是正式文件保存主體。

ERP 負責：

```text
查詢
統計
提醒
索引
彙整
列印
交接
```

記帳士負責：

```text
正式會計
科目判斷
申報
報稅
財務報表
```

一句話：

```text
先讓車行營運不亂，再談完整會計自動化。
```
