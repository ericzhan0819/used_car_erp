## 1. 本文件目的

本文件定義 `P1-MVP-UX-OPS-2：Guided Business Flow Forms` 的產品方向與 UX 邊界。

本階段目標不是繼續把更多功能塞進 `Used Car Vehicle` 標準 DocType 表單，而是把業務主流程逐步整理成：

```text
任務卡片式業務操作
```

也就是：

```text
一張卡處理一件事
一個任務只問必要欄位
業務只輸入商業事實
系統在背景處理資料流
會計文件與會計判斷留在會計作業
```

---

## 2. 核心 UX 原則

業務端整體風格採用：

```text
像任務卡片，一張卡處理一件事
```

每張任務卡應具備：

- 明確任務名稱
- 清楚下一步按鈕
- 只顯示本任務需要填的欄位
- 不顯示會計術語
- 不顯示 ERPNext 技術文件連結
- 阻擋性錯誤才使用 banner
- 一般提示放欄位旁或卡片內小字

業務端不應該像完整 ERP 表單。  
業務人員不應被要求理解底層 DocType、會計文件、會計候選、借貸方或 ERPNext 技術欄位。

---

## 3. 業務與會計邊界

本階段繼續沿用既有產品邊界：

```text
業務歸業務
會計歸會計
```

### 3.1 業務人員負責

業務人員只輸入真實世界發生的商業事實：

- 這台車是什麼車
- 向誰買
- 買多少錢
- 買入來源是什麼
- 有哪些監理 / 稅務注意事項
- 花多少錢維修 / 美容 / 代辦 / 拍場
- 賣給誰
- 成交價是多少
- 收多少訂金
- 收多少尾款
- 何時交車

### 3.2 系統負責

系統在背景處理：

- 建立車輛資料
- 建立 / 綁定必要 ERPNext 資料
- 完成入庫
- 切換車輛狀態
- 建立金流紀錄
- 建立傳票草稿
- 更新收支摘要
- 檢查金額一致性
- 推導會計作業待辦
- 推導管理摘要

### 3.3 會計人員負責

會計人員只在「會計作業」處理：

- 審核傳票草稿
- 確認入帳
- 處理正式 Journal Entry
- 處理 Sales Invoice
- 處理預收款沖轉
- 處理正式會計閉環

## 4. 業務頁禁止 / 避免字詞

業務頁避免出現：

- Sales Invoice
- Journal Entry
- Voucher Draft
- Money Flow
- GL Entry
- 借方 / 貸方
- 會計科目
- 預收款沖轉

這些字詞只應出現在：

- 會計作業
- 技術文件
- 內部開發文件
- 管理 / 會計專用頁面

車輛頁、總覽業務入口與任務卡片不應暴露這些術語。

## 5. 新增 / 買入車輛流程

### 5.1 按鈕名稱

車輛主流程按鈕名稱：

```text
新增車輛
```

總覽快捷入口可顯示為：

```text
新增買入車輛
```

### 5.2 流程形式

新增車輛不應一次顯示所有欄位。  
應拆成至少兩步：

```text
Step 1：車輛基本資料
Step 2：收購資料
```

車輛基本資料與收購資料不要在同一頁輸入。  
Step 1 完成後，透過「下一步」進入 Step 2。

### 5.3 Step 1：車輛基本資料

最少欄位：

- 廠牌
- 車型
- 年式
- 車牌
- VIN / 車身號碼
- 里程
- 顏色

完成後按：

```text
下一步
```

### 5.4 Step 2：收購資料

最少欄位：

- 購車價
- 買入來源
- 供應商 / 原車主
- 收購業務
- 監理 / 稅務旗標 checklist

買入來源預設：

```text
個人
```

買入來源選項：

- 個人
- 同行
- 拍賣場
- 其他

監理 / 稅務旗標呈現方式：

```text
全部 checkbox 顯示，讓業務自己勾
```

### 5.5 完成後系統動作

建立車輛後，系統應自動：

```text
建立車輛
→ 完成入庫
→ 狀態進入整備中
```

若必要欄位不足，例如 VIN、購車價、買入來源缺漏，應阻擋送出並明確提示缺少什麼。

### 5.6 本流程不顯示

新增 / 買入車輛任務卡不應顯示：

- Sales Invoice
- Journal Entry
- Voucher Draft
- Money Flow
- 會計科目
- 借方 / 貸方
- 預收款沖轉
- ERPNext 技術文件連結

## 6. 整備中支出流程

### 6.1 任務入口

車輛狀態為整備中時，業務應看到明確任務按鈕：

```text
新增支出
```

首頁也可保留快捷入口：

```text
新增支出
```

### 6.2 支出表單最少欄位

- 支出日期
- 支出類型
- 金額
- 付款方式
- 付款對象
- 備註

### 6.3 支出類型

保留：

- 維修支出
- 美容支出
- 代辦支出
- 拍場支出
- 其他支出

不列入本階段主要支出類型：

- 整備支出

原因：整備是一個流程狀態，不一定要作為支出分類；實際支出應更精確分為維修、美容、代辦、拍場、其他。

### 6.4 新增後顯示

新增支出後，業務端應看到：

```text
顯示在收支摘要，但不顯示會計狀態
```

不可顯示：

- Money Flow
- Voucher Draft
- Journal Entry
- Sales Invoice
- 會計科目
- 借方 / 貸方

### 6.5 底層允許行為

底層仍可建立：

- `Used Car Money Flow`
- `Used Car Voucher Draft`

但業務端不應看到這些技術名稱。

業務端語意應為：

```text
支出已記錄
收支摘要已更新
```

## 7. 整備完成 / 上架流程

### 7.1 按鈕名稱

```text
整備完成並上架
```

### 7.2 上架表單欄位

- 底價
- 開價
- 上架日期
- 銷售備註

### 7.3 權限規則

底價與開價只能主管填。

其中：

```text
底價只有主管看得到
```

業務人員不可看到底價。  
一般業務視角只應看到可銷售資訊，不應看到主管底價或內部底線。

### 7.4 完成後系統動作

按下確認後：

```text
車輛狀態 → 上架中
```

### 7.5 待後續定義

本文件先記錄方向，後續實作前仍需確認：

- 開價是否也完全主管限定
- 業務是否能看到開價
- 是否需要「待主管填價」狀態
- 是否需要主管專屬 Dashboard 卡片

## 8. 保留 / 收訂金流程

### 8.1 表單欄位

- 客戶
- 訂金金額
- 收款日期
- 付款方式
- 成交價
- 銷售人員

### 8.2 收訂金後狀態

使用者偏好：

```text
依訂金是否入帳決定
```

產品語意需進一步定義為：

- 業務送出訂金資料後，業務端可顯示「已收訂金」
- 會計是否入帳不直接暴露給業務
- 車輛是否進入「保留中」需依內部規則決定

### 8.3 業務端會計狀態顯示

業務端不需要看到訂金是否已由會計確認。

顯示方式：

```text
只顯示已收訂金
```

不可顯示：

- 待會計入帳
- Voucher Draft
- Journal Entry
- 預收款沖轉

### 8.4 待後續定義

「依訂金是否入帳決定」目前仍需轉成明確產品規則。  
後續至少需要在以下兩種方向中選擇：

```text
A. 業務輸入訂金後立即進入保留中，會計狀態只在會計作業處理。
B. 業務輸入訂金後顯示已收訂金，但主管 / 會計確認後才進入保留中。
```

本文件先保留此問題，不在 Step 1 直接定案。

## 9. 售出 / 收尾款流程

### 9.1 表單欄位

- 客戶
- 成交價
- 尾款金額
- 收款日期
- 付款方式
- 交車日期
- 銷售人員

### 9.2 尾款與確認成交關係

使用者偏好：

```text
依情境可選
```

因此售出流程應允許兩種路徑：

```text
路徑 A：收尾款並確認成交
路徑 B：先記錄尾款，再另外確認成交
```

具體 UI 可在後續任務中再拆分。

### 9.3 金額一致性檢查

系統應背景檢查：

```text
訂金 + 尾款 = 成交價
```

若不相等，需跳出提示：

```text
訂金 + 尾款 與 成交價不一致
差額：{difference}
```

使用者必須二選一：

```text
返回修改
或
輸入原因後繼續送出
```

若沒有輸入原因，不允許送出。

### 9.4 成交後業務頁顯示

成交後業務頁只顯示：

- 成交價
- 訂金
- 尾款
- 支出摘要

不顯示：

- 會計文件
- 會計狀態
- Journal Entry
- Sales Invoice
- 預收款沖轉

## 11. 提示 / Banner 規則

### 11.1 Banner 使用原則

只有邏輯阻擋時才顯示 banner。

例如：

- 必填欄位缺漏
- 金額不一致
- 車輛狀態不允許此操作
- 有監理 / 稅務風險阻擋流程

### 11.2 一般提示

一般提示不放頁首 banner。

應改為：

- 欄位旁小提示
- 卡片內說明文字
- 按鈕旁輔助文字

### 11.3 風險提示

風險使用紅色或橘色。

例如：

- 禁止異動
- 有未清償貸款
- 有欠稅 / 罰款
- 金額不一致
- 缺必要文件

## 12. 15-1 與稅務邊界

本階段延續既有稅務邊界：

```text
15-1 只用在售車營業稅估算。
購入成本 = 購車價。
整備費、維修費、美容費、拍場費、代辦費，不併入 15-1 的中古車購入成本。
```

### 12.1 採購價語意

`purchase_price` 固定為：

```text
購車價，不包含整備、維修、美容、拍場費、代辦費或其他後續費用。
```

### 12.2 管理損益

管理損益可以包含後續支出：

```text
成交價
- 購車價
- 維修費
- 美容費
- 拍場費
- 代辦費
- 其他直接支出
= 管理毛利
```

### 12.3 15-1 營業稅估算

15-1 不使用整備費等後續支出作為購入成本。

估算邏輯為：

```text
售車銷項稅額 = 售車價 ÷ 1.05 × 5%
15-1 購入可扣抵稅額 = 購車價 ÷ 1.05 × 5%
實際可扣抵 = min(購入可扣抵稅額, 銷項稅額)
預估本車營業稅 = 銷項稅額 - 實際可扣抵
```

- 大量 accounting runtime 改動

- 完整 dashboard 重構

- AI 操作層
- 官網同步
- Odoo / Laravel 整合

每一步都應保持小步、可測、可回滾。

## 16. 最終結論

`P1-MVP-UX-OPS-2` 的方向不是把 ERPNext 原生表單變好看。

真正目標是：

```text
讓業務不需要知道 ERPNext 與會計文件存在。
```

業務端只需要依照任務卡片完成：

```text
新增車輛
新增支出
整備完成並上架
收訂金
收尾款
確認成交
查看收支摘要
```

系統與會計作業負責承接後續資料流。
